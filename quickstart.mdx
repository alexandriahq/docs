Ambient is Alexandria's open-source data-mining runtime for enterprise laptops. It captures reasoning and action traces from high-stakes computer work, turns them into local structured context, and helps agents understand long-running human tasks.

Ambient currently ships as two desktop apps:

- **Ambient App** — the local Electron app that owns capture, context workers, the encrypted `ambient.sqlite` database, memory, intents, and handoffs.
- **Ambient Bridge** — the companion app that owns WorkOS sign-in, local pairing, diagnostics, and the only public-network egress path for inference.

<Note>
  Ambient is local-first. Ambient App does not talk directly to the public internet. When remote inference is used, Ambient talks to Bridge over authenticated OS IPC; Bridge talks to Alexandria Server; the server forwards encrypted requests to Tinfoil.
</Note>

<Steps>
  <Step title="Get the release links">
    Use the product-scoped release links provided by your Alexandria workspace, deployment admin, or release channel.

    You need both products:

    - `ambient-app` — Ambient App
    - `ambient-bridge` — Ambient Bridge

    For experimental builds, use the exact versioned links from the release comment or deployment note. Avoid moving "latest" links for experimental Windows builds.
  </Step>
  <Step title="Install both desktop apps">
    On macOS, install both apps from their `.dmg` files and launch them from **Applications**.

    For realistic capture and permission testing, use a signed/notarized build. macOS permissions are tied to the app bundle identity and code signature, so ad-hoc or dev builds can lose grants after rebuilds.
  </Step>
  <Step title="Sign in to Ambient Bridge">
    Open **Ambient Bridge** first and sign in through the browser.

    Bridge uses WorkOS for account auth and stores the sealed session in the OS keychain. Keep Bridge running; Ambient App uses it for account state, pairing, and inference egress.
  </Step>
  <Step title="Open Ambient App and pair with Bridge">
    Open **Ambient App**.

    Ambient discovers Bridge through a signed local descriptor and connects over OS IPC. If Bridge asks you to approve a pairing request, approve it to let Ambient App make authenticated local requests.
  </Step>
  <Step title="Choose what Ambient can capture">
    During setup, choose the local signals Ambient should use.

    The current capture policy supports sources such as screenshots, microphone audio, system audio, keyboard input metadata, active app/window metadata, and accessibility-derived context. Screenshots are currently treated as a required source in setup so Ambient can understand visible work.

    You can also exclude private apps, system-audio sources, and browser URL patterns before capture runs.
  </Step>
  <Step title="Grant system permissions">
    Ambient App owns capture permissions. Grant the macOS permissions requested by the setup flow.

    Depending on enabled sources, this may include **Screen Recording**, **Microphone**, **Accessibility**, **Input Monitoring**, and system-audio related access. Some grants require restarting Ambient App before capture can work.
  </Step>
  <Step title="Verify capture is running">
    Open Ambient App and check the capture/status surfaces.

    The **Feed** should begin showing recent context chunks after capture and context processing run. The **Permissions** and capture settings surfaces show whether sources are healthy, blocked, or not producing artifacts.
  </Step>
  <Step title="Review memory and handoffs">
    Use the **Memory** view to inspect the local Markdown memory graph, and use **Handoffs** to review agent-ready work.

    A handoff is a structured plan tied to an inferred intent. You can review its steps, refine them, deny the handoff, or accept it and open the configured agent flow.
  </Step>
</Steps>

## What Ambient Is Building

Ambient is not a generic screen recorder or time tracker. It is a local memory and task-mining layer for long-horizon work.

The system turns captured activity into chunks, observations, intents, memory, and handoffs so agents can reason from the user's real working timeline instead of starting from an empty prompt.
